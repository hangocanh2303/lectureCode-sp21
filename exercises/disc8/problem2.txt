(a) Here are three potential implementations of the Integer’s hashCode() function.
Categorize each as either a valid or an invalid hash function. If it is invalid,
explain why. If it is valid, point out a flaw or disadvantage. For the 2nd
implementation, note that intValue() will return that Integer’s number value
as an int.
public int hashCode() {
    return -1; // valid but theta(N) -> one bucket hash
}
public int hashCode() {
    return intValue() * intValue(); // valid but increase collisions
}
public int hashCode() {
    return super.hashCode(); // invalid because dependent hashCode of super class, worst case -> hashCode use memory address -> wrong
}

(b) For each of the following questions, answer Always, Sometimes, or Never.

1. If you were able to modify a key that has been inserted into a HashMap
would you be able to retrieve that entry again later? For example, let us
suppose you were mapping ID numbers to student names, and you did a
put(303, "Anjali") operation. Now, let us suppose we somehow went to
that item in our HashMap and manually changed the key to be 304. If we
later do get(304), will we be able to find and return "Anjali"? Explain.


Answer:
Sometimes: Trong đa số trường hợp khi get(304), mã băm sinh ra sẽ khác bucket (vì Anjali thực tế nằm trong bucket của 303, dù đã sửa nó thành 304 nhưng thực tế vẫn nằm trong bucket 303)

2. When you modify a value that has been inserted into a HashMap will
you be able to retrieve that entry again? For example, in the above
scenario, suppose we first inserted put(303, "Anjali") and then changed
that item’s value from "Anjali" to "Ajay". If we later do get(303), will
we be able to find and return "Ajay"? Explain.

Answer: always
hàm băm tính toán dựa trên key, không ảnh hưởng tới value
